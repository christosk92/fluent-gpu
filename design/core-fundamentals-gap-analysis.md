# FluentGpu — Core Fundamentals Gap Analysis

*What FluentGpu's design is missing in the core fundamentals, measured against how React (Fiber/lanes/hooks), Flutter (3-tree/constraints/arena/semantics), Compose/SwiftUI/Solid (the no-vdom camp), and UI-framework design practice actually work. Produced from web-backed research + a two-lens gap analysis (programming-model + platform), each adversarially reviewed (6 false gaps removed, 6 real ones added).*

## 1. Executive Summary

**Verdict: FluentGpu's core is sound — and unusually complete for a v1.** The diff/hooks/effects engine is essentially React-complete, correctly trades against the signals camp, and keeps the one asset (a pure, restartable render phase publishing an immutable `SceneFrame`) that the signals camp would forfeit. Do not migrate to signals or a slot table. The real work is not in the diff engine; it is concentrated in two clusters: **(a) the concurrent-era scheduling/async-UX primitives React added *on top of* exactly FluentGpu's model**, and **(b) the interaction/semantics obligations a draw-everything engine inherits** because no native control sits behind anything.

The 5 most important gaps, in priority order:

1. **Scheduling: priority lanes + automatic batching** — the two reserved-but-unbuilt phase-3 update-queue decisions. FluentGpu has a *scalar*, system-driven `RenderPriorityPolicy` and an in-place setState applied at call time. Both are API-shape decisions painful to retrofit once hooks/coalescer APIs are in use. (Tier 1)
2. **Suspense-style async boundary** — a declarative atomic-reveal/keep-stale primitive vs today's ad-hoc per-call-site skeletons. The single biggest *programming-model* gap; it's a reconciler concept, not a bolt-on hook. (Tier 1)
3. **External-store consistency contract** (`useSyncExternalStore`-analog) for `UseObservable`/`UseResource` — dormant in v1's single-pass model, latent the moment slicing flips on, and a breaking hook-shape change to retrofit. (Tier 1)
4. **Interaction obligations: gesture arena, text-selection seam (on-screen + `ITextRangeProvider`), overlay/portal manager** — the obligations a rasterized engine owns; their *substrate* exists but the *composing contracts* don't, and they bake into a control library's assumptions. (Tier 1)
5. **RTL layout mirroring** — the box layout is direction-blind (`FlowDirection` lives only in text shaping). Structurally hard to retrofit because every alignment/margin/flex-direction site must consult writing direction. (Tier 1)

Everything else (selectors/derived state, springs+retarget, optimistic updates, UIA collection relations + virtualized realization, control kit, i18n formatting, cursor contract, edge-autoscroll) is real but Tier 2/3 — fold soon or defer. The through-line: **FluentGpu built the hard substrates; what's missing is the composing contracts and the concurrent-era completeness — and several of those are cheap to *stamp* now and expensive to *retrofit* later.**

## 2. How the Field Works (the bar)

**React (Fiber / lanes / hooks / concurrent).** A view tree of elements is diffed into a retained fiber (host) tree. The render phase is **pure and interruptible**; commit is the only side-effecting phase, split into before-mutation → mutation (detach refs, layout-effect cleanups) → swap current → layout (attach refs, run layout effects) → paint → passive, run **bottom-up, cleanup-before-create**. The concurrent era added, *on top of this same model*: **lanes** (a 31-bit bitmask decoupling priority *ordering* from *batching*, replacing the older scalar `expirationTime`); **automatic batching** of all setStates within an event handler *and* across async continuations into one render; **`startTransition`/`useTransition`/`useDeferredValue`** to mark updates non-urgent; **`<Suspense>` + `use(promise)`** for declarative async boundaries with keep-stale-during-transition; **`useSyncExternalStore`** (snapshot + pre-commit re-check + demote-to-blocking) for tearing-safe external reads; **`useOptimistic`/`useActionState`** for optimistic UX. Positional hooks; identity = (type, key) at a position; type/key change ⇒ unmount+mount with state destroyed.

**Flutter (3-tree / constraints / arena / semantics / draw-everything).** Widget (config) → Element (lifecycle/identity) → RenderObject (layout/paint). Layout is **constraints-down, sizes-up**, with intrinsic sizing as a cached escape hatch, baselines and writing-direction carried as **protocol data** (RTL mirrors automatically). Because it owns the pixels, it *owns the four obligations the OS otherwise gives for free*: the **semantics tree** (a merged/compacted `SemanticsNode` projection, gated when no AT is listening), **text interaction** (`RenderEditable`: caret, transactional IME, selection handles, the AT text pattern — Levien's "litmus test"), **overlays** (focus containment/restore, z, light-dismiss), and **gesture disambiguation** via the **`GestureArenaManager`** ("first to accept or last to not reject wins," eager-win, pointer-up sweep, hold/release for double-tap, `GestureArenaTeam` for selection).

**Compose / SwiftUI / Solid (the no-vdom camp).** They delete the per-render allocation by tracking dependencies at read-time and mutating an in-place cache: **Compose** = a compiler-plugin-driven **slot table** with positional memoization + strong-skipping stability + `derivedStateOf` (push-eager/pull-lazy edge-collapse) + MVCC snapshots that solve tearing by isolation; **SwiftUI** = an **AttributeGraph** with property-granular `@Observable` subscription; **Solid** = a **run-once signal graph** — "components are pure overhead," recompute only the reactive leaf that changed. The cost: a bespoke runtime/compiler, call-site identity footguns, and — decisively — **forfeiting the pure render phase** as a lever for off-thread/interruptible work.

**The draw-everything obligations** (the bar a rasterizing engine is held to): a lockstep semantic tree with collection relations and virtualized realization; full text interaction; overlays with focus+z+dismiss; gesture arbitration; cursor/pointer feedback; motion that is interruptible/retargetable; RTL *layout* (not just glyph) mirroring; locale formatting.

## 3. The Big Fork: VDOM+Reconcile vs Signals

**Verdict: keep vdom+reconcile. Do not add a signals layer or a Compose slot table in v1.** Steal exactly one idea (a derived/selector node). The reasoning, specific to FluentGpu:

1. **The churn the signals camp deletes is *Element construction*, not the diff.** The Gen0 source is `new Element` per render. FluentGpu's 3-signal memo skip (`SelfTriggered || propsChanged || HasConsumedContextChanged`) already skips `Render()` on clean components — and thus the construction of *that subtree*. The construction that remains is for components that genuinely re-rendered: exactly the set a signal graph would also recompute (this is Xilem's `Memoize`). A signals migration buys *less* than implied.

2. **A signals/effect-writes-the-scene model forfeits the pure, restartable render phase** — the single precondition for FluentGpu's *designed* time-sliced/off-thread reconcile and its atomic-on-complete PUBLISH seam (immutable `SceneFrame` snapshot). Adding fine-grained signals throws away the very thing that keeps the residual fixable later.

3. **Compose's slot table requires a bespoke compiler plugin + stable call-site identity + a strong-skipping stability discipline.** FluentGpu would be trading a working, predictable, source-gen-based model for a from-scratch positional-memoization runtime under NativeAOT — unjustified complexity at v1.

**Correcting the residual framing (per critique).** The painpoints "irreducible per-render Element churn" label is too strong as stated. It is **irreducible *per genuinely-rendered component*; aggregate churn is bounded by the 3-signal skip — which is a *maintained* property, not a free one.** Without memoization discipline it is reducible-in-aggregate and "can be far worse than implied." The honest statement for the spec: *the model's edge allocation is per-rendered-component Element construction; the 3-signal skip bounds it in aggregate, and maintaining that discipline (props equality, context-projection granularity) is a standing obligation, not a one-time win.* The keep-vdom verdict is unchanged and correct.

**The one idea worth stealing — and it's already half-built.** Compose's `derivedStateOf`: a notify-only-when-result-*changes* node. FluentGpu drives chrome from high-frequency inputs (`ScrollOffset`, `AnimTrack.DrivenClock`, pointer, `ISystemColors.Epoch`). It already ships the load-bearing instance of this: the **boxless value-context selector** — `Context<T>.ToDepKey(T)` projection + the deliberate `Context<uint>`-over-`ISystemColors.Epoch` pattern, where a 112-byte `SystemTint` is intentionally *not* the context payload (consumers subscribe to a coarse `uint` version and re-read the fat value on demand). That **is** "subscribe to a projection, re-read on demand" for the highest-frequency input the engine has, and it is *ahead of* stock React. What's genuinely absent is the **generalization**: a `UseDerived`/`UseContextSelector` over arbitrary projections so app code doesn't hand-roll the `Epoch`-as-`uint` trick per call site. (Tier 2; one primitive serves §6 and the value-collapse half of §4.) **The migration question itself is settled / out-of-scope (Tier 3).**

## 5. Tier-1 Deep Dives

### 5.1 The phase-3 update queue: lanes (P1) + automatic batching (P2a)

These are one decision with two faces, and both are *reserved-but-unbuilt* in the corpus — phase 3 is explicitly "a reserved no-op; v1 applies in place at call time." That stub is the single most consequential API-shape decision left open, because the *shape* of the update queue (per-update enqueue carrying a lane, vs in-place mutate) cannot be changed once app code and hooks depend on the timing.

**Direction.** Replace the in-place phase-3 with a real update queue:
- Each `setState` enqueues an `(fiber, updater, lane)` record rather than mutating in place. Lane is derived from **update cause**: input-handler-originated = urgent; data-arrival / `await`-continuation / `startTransition`-wrapped = transition. Use a small bitmask (even 8 lanes), not a scalar — the scalar `RenderPriorityPolicy` is precisely the `expirationTime`-era design React abandoned because it conflates ordering with batching.
- **Automatic batching falls out for free**: drain the queue once per coalesced frame, so all setStates in a handler *and* across an `await` boundary fold into one consistent render. This is the React-18 batching semantics the corpus today does not provide (frame-coalescing batches *frames*, not *updates within a handler*).
- `RenderPriorityPolicy`'s 16ms budget stays — but as the *executor* of lane scheduling (which lanes to flush this frame), not as the priority *source*.
- **Even if interruptible execution (P4) ships later, stamp lanes now.** Stamping is the cheap part; retrofitting a lane field onto an in-use update path is the expensive part.

### 5.2 Suspense boundary (P2b)

The keep-stale *capability* exists per call-site (`ComponentAnchor` stable identity, synchronous-cleanup→`DetachedAnim` keep-alive, `CrossFadeImage`). What's missing is the **boundary abstraction**: which subtree swaps atomically, where the fallback mounts, and how reveal interacts with transitions. This is a reconciler+layout concept, not a hook.

**Direction.** A `Boundary`/`Suspense` *element* the reconciler understands: when a descendant `UseResource` is pending, the reconciler mounts the boundary's fallback as a unit; on ready, it atomically swaps to content. Support **nested** boundaries (progressive reveal) and a **transition-aware keep-stale** rule — during a transition (P1 lane), keep already-revealed content visible rather than flashing the fallback. Build the keep-stale on the existing `DetachedAnim` slab so committed visuals survive the swap without relayout. This composes with P1 (the keep-stale rule *is* lane machinery) and P7 (optimistic).

### 5.3 External-store consistency (P3)

Dormant in v1's single-pass model, latent the instant slicing flips on. The fix is a *hook contract shape*, which is why it must be decided now. The corpus already ships the correct shape for one store: `ISystemColors` is volatile-published, written only on OS change, read by-value-on-demand via a stable instance with an `Epoch` change-trigger — i.e. it already has a getSnapshot/version pair.

**Direction.** Generalize that exact pattern into the `UseObservable`/`UseResource` contract: require a `(subscribe, getSnapshot)` pair returning a referentially-cached, immutable snapshot taken at frame start; reserve a **pre-PUBLISH re-check** that, under slicing, demotes to a blocking single-pass frame on snapshot mismatch (`useSyncExternalStore`'s tear-detect). The narrow, true gap: arbitrary live `IObservable`/model objects lack this snapshot+version contract — system colors already have it.

### 5.4 Gesture arena (L2) + text-selection seam (L1/L3)

The arena is load-bearing under *both* future controls and text selection. The honest scoping correction: it is **not "purely additive."** A real arena changes *when* a recognizer commits — capture becomes **tentative until arena resolution**, resolution defers across pointer-move frames (eager-win / pointer-up sweep / hold-release), and the `e.Handled`-over-single-captured-route model becomes one input *into* arena resolution rather than the resolution itself. Keep `PointerFsm` as the per-recognizer implementation; add the arena as the coordinator that owns tentative capture.

Design L1/L3 **together**: the same `TextLayoutSlot` read-side (`GetSelectionRects`/`HitTestTextPosition`) must back on-screen selection highlight *and* `ITextRangeProvider`. Add `SelectionState` (anchor/extent/affinity) as a scene column with a `DrawSelectionRectCmd` overlay opcode; wire selection-drag through the arena (it needs the `GestureArenaTeam` pattern), with the **edge-autoscroll driver (L11)** writing `ScrollOffset`. Define the editable transactional buffer seam (`ITextStoreACP2`-shaped) now even though Imm32-v1 fills it and editing is v2 — so controls and the UIA Value/Text patterns wire against a stable contract.

### 5.5 RTL layout mirroring (L5)

Structurally hard to retrofit because every alignment/margin/flex-direction site must consult writing direction. The safe form, given the corpus's binding golden-parity constraint: **resolve logical→physical at the reconciler's `WriteLayout` boundary** (where `LayoutInput` is already written and `DiffProps` computed), so the ported Yoga `CalculateLayoutImpl` stays physical and bit-for-bit with the ported Reactor-Yoga numerics gate. Mirroring inside the ported core would threaten that verbatim-parity contract. Add `FlowDirection` as inherited context resolved at `WriteLayout`; make flex direction, justify/align, and `Edges4` resolve start/end→physical there.
