# Wavee Backend Architecture — Consolidated Gap Report

This is the de-duplicated, cross-domain worklist for `docs/plans/wavee-native-backend-architecture.md`. Eleven domain reviews converged on a small set of structural holes that recur across many sections; they are consolidated below. Evidence paths are under `C:\WAVEE\WaveeMusic\src` and `C:\WAVEE\fluent-gpu\app\Wavee.Core` unless noted.

---

## 1. MISSING SUBSYSTEMS (whole things absent from the doc)

**A. Offline download manager.** The doc only *reads* `audio_cache.complete` (§7.4); nothing represents user *download intent*. Needs: a `pinned_downloads(uri, container_uri, quality, state, …)` table distinct from the opportunistic cache; a headless fetch driver (the existing `ProgressiveDownloader.FetchRangeAsync` only runs bound to an *actively playing* track — `playback.md:433`); concurrency caps + Wi-Fi-only/charging/metered gating; eviction-immunity + a separate quota (today `PruneCacheAsync` evicts strictly by `LastAccessed` against one budget — `AudioCacheManager.cs:284`, so a pin is silently pruned); per-download quality tier chosen *at download time* (bytes are immutable once landed — hardcoded `OGG_VORBIS_320` today); and **partial-download resume** — `audio_cache.complete INT` cannot represent the per-file `CachedChunks` bitmap `AudioCacheManager` already maintains (`LoadExistingEntries`, lines 347-391). **Critically, this subsystem owns the AudioKey-on-disk requirement (see B).** Store/seam impact: a `DownloadState` field on entity rows (`Models.cs` has none), download progress signals, a per-container "downloaded iff all members complete" rollup. Episodes are a *separate* lifecycle: `podcast_episodes(download_state, downloaded_bytes, local_file_path)` already exists in WaveeMusic (`MetadataDatabase.cs:957`) and §7.4 covers only `track`/`local`.

**B. Persisted AudioKey table.** `audio_cache.complete` is not a sufficient offline-playability predicate. Decrypt requires the 16-byte AudioKey; `AudioKeyManager` persists AP keys to SQLite "valid forever (FileIds immutable)" but keeps PlayPlay-fallback keys **in-memory only** (`AudioKeyManager.cs:145,339-343`). The §4.1 schema has no `audio_keys(file_id, key)` table and §6.1 frames the key as an in-memory cache. Consequence: a fully-cached file is undecryptable after restart, and a PlayPlay-keyed track is un-playable offline *even fully downloaded*. The offline-playable predicate must be `complete ∧ AP-key-on-disk`, and the download manager must force the AP-key path.

**C. Play-history / recently-played store.** No `play_history` table, populator, or read path exists in §4.1, yet it is the source of `Track.PlayCount` (`Models.cs:84`, stored nowhere), the `RecentlyPlayed`/`MostPlayed` library sorts (`PlayHistoryEntry.cs:229`), `LibraryStats.TotalListeningTimeMs`, the Home "Pick up where you left off" hero slot (`RecentlyPlayedService`, fed by both Home parse *and* live `PlaybackContextChangedMessage`), and the gabo context §6.1 itself warns "silently breaks." It needs an append/ring table with a **retention policy** (the §4.6 string-arena eviction says nothing about history rows), a playback-completion write path, and `episode_progress(uri, locale, position)` for cross-device resume (`Episode.ProgressMs`, `Models.cs:126`).

**D. gabo / playback-reporting event subsystem.** The doc reduces gabo to "the anti-fraud context block" and folds reporting into a *pure* reducer (§7.2) — but `Connect/Events/` is a full side-effecting pipeline: `EventService` + ~10 `IPlaybackEvent` types, `PlaybackMetrics` with interval tracking straddling pause/resume/seek (`PlaybackOrchestrator.Metrics.cs:149,285`), per-play `PlaybackId`, `PlaybackReason` (`TrackDone/FwdBtn/ClickRow/Remote`) for both `reason_start`/`reason_end`, a per-stream monotonic `_sequenceId`, and the *only working transport* (mercury/HTTPS both 404). A pure reducer has no seat to emit an ordered, interval-bracketed durable side-effect stream; it needs its own emitter actor (the §8 outbox is for library writes). Private-session `SuppressPlayHistory` gates this stream.

**E. Search + offline-search index.** The §4.1 `entities(uri,kind,blob,…)` table is an opaque blob — `LIKE`/`MATCH` cannot reach names, so the locked "browse/search *any* cached container fully offline" promise is physically impossible. WaveeMusic's real `entities` carries first-class `title/artist_name/album_name/publisher` columns with `COLLATE NOCASE` indexes (`MetadataDatabase.cs:1263`; note: it uses indexed columns, **not** FTS5). Online search is *two* undesigned Pathfinder pipelines — debounced `topResultsV2` suggestions vs. full per-type paged ops (`SearchService.cs:37`) — plus top-result ranking, `SearchSectionEntity` grouping, `matchedFields` highlight, and `AggregateCatalog` cross-source merge/dedup. The seam `SearchResults(Tracks,Albums,Artists,Playlists)` (`Library.cs:7`) drops Episodes/Shows/Users/Genres/top-result. Also missing: a `Resolve(spotify:uri)` seam for paste-to-resolve, and durable cached-suggestion + recent-searches storage (server-invalidated online — `IPathfinderClient.cs:201`).

**F. Home / Browse / discovery pipeline.** No `home_feed` table anywhere; "Home" is only a capability flag. `PageCache` holds a single RAM field, never persists (`PageCache.cs:16`) — so there is no cold-launch or offline Home, violating tenet 1 for the *first screen*. Needs `home_feed(account, facet, blob, fetched_at, ttl)` (Home has no stable URI/etag/snapshot_id — it cannot ride the `entities` path), a poll-the-whole-response TTL model (no delta, 5-min `PeriodicTimer`, suspended during playback), the real `HomeFeedCache.ApplyDiff` semantics (section reorder, ContentType-drift forcing card Replace, sparse-response display-data carry-forward — the same template-selector-at-recycle hazard transfers to the fluent-gpu virtualizer), per-card *baseline enrichment* as a distinct fetch shape (`GetFeedBaselineLookupAsync` → canvas video/audio-preview/hero-color, none on seam `HomeCard`), facet/chip parameterization, and a whole separate **Browse** editorial grid (`GetBrowseAllAsync`/`GetBrowsePageAsync`). The V1/V2 trait `JsonExtensionData` re-deserialize is exactly the §10 AOT risk.

**G. Pathfinder / GraphQL view-plane cache.** §6.1 reduces ~30+ Pathfinder operations (`ArtistOverview`, discography, credits, concerts, similar, `TrackCredits`…) to a one-line "parse JSON, project" with no cache tier, key, or schema. The proto-per-`(uri,ext)` `extension_cache` cannot key by `(persisted-query-hash, variables)` nor store relationship-shelf edges. **Sub-gap:** persisted-query hashes rotate server-side, so the cache key must carry an operation/app version or it decodes payloads against a stale schema after a hash bump. Right-panel Details over-fetch (credits/concerts re-fetched every panel open) collapses into this.

**H. Playlist folders / rootlist tree.** The flat `playlists`+`playlist_items` schema has no rootlist entity, no folder model, no group-marker column. The sidebar is a tree parsed from `spotify:user:{u}:rootlist` interleaving playlist URIs with `start-group`/`end-group` markers; folders have **no fetchable URI** (`RootlistTreeBuilder.cs`, `folder_path` at `MetadataDatabase.cs:1014`). Reordering the sidebar / moving a playlist between folders is a **distinct rootlist Op** — a second OpRebase write target §8.5 never models — delivered live via `RootlistModificationInfo` dealer pushes needing its own subscription lane.

**I. Connect device-announce (PutState) + the dealer REQUEST→reply third primitive.** §6.1 declares "exactly two primitives." Neither can express a server→client `REQUEST` demanding a correlated reply on the same WS within ~10 s (`ConnectCommandHandler.cs:230`) or Spotify 502s the device. Also entirely absent: `PutStateRequest`/`DeviceInfo`/`Capabilities` to `connect-state/v1/devices` — without it Wavee never appears in any device picker and every command is refused. Two coupled sub-gaps: the reply is *ack-of-receipt only* (real result rides a follow-up PutState — `:233`), and every PutState must embed a live `PlayerState` pulled from the Playback reducer (`BuildCurrentPlayerState`, `DeviceStateManager.cs:248`) — a mandatory back-edge the clean populator/playback split has no channel for.

**J. Per-account store partitioning.** `CredentialsCache` is explicitly multi-user (`{username}_credentials.dat`, `CredentialsCache.cs:225`), but §4.1 is a single URI-keyed DB with only `meta.account`. Nothing partitions `entities`/`collection_items`/`playlists`/`outbox`/`audio_cache` per account — logging A→B replays A's outbox under B, shows A's library, and lets B play A's decrypted cached audio (privacy + correctness bug). Needs per-account DB files (or an account column on every table) + a clear-on-switch contract.

**K. Collaborative-playlist permission model.** `playlist_permission.proto` defines `PermissionLevel`, `Capabilities{can_edit_items,…}`, `SetPermissionLevelRequest`; `SelectedListContent` carries `capabilities(18)`; the UI gates every mutation on these (`PlaylistMutationCoordinator.cs`). §8.5 frames editing as "a playlist you own," erasing CONTRIBUTOR writes, capability-gated mutation, per-item `added_by`, and genuine two-writer conflict — for which "local-intent-wins" has no answer.

**L. client-token subsystem.** §6.1 wrongly lists "login5" as a `RequestAsync` caller; login5 and client-token are **HTTPS side-channels**, not Shannon-AP callers (`Login5Client` over plain `HttpClient`, `SessionData.cs:65`; `ClientTokenManager` POSTs `clienttoken.spotify.com`). The doc has no seat for the client-token at all: hashcash-PoW-solved, ~5-min refreshed, required header on every spclient call, carrying a server-allowlisted version string whose staleness gets gabo events rejected (`ClientTokenManager.cs:25,259`).

**M. Dead-letter / optimistic-rollback for dropped outbox ops.** At `MaxRetries=10` the row is DELETEd (`OutboxProcessor.cs:70`) while the store still shows `sync=Pending`/saved — the UI shows "saved" forever until an unrelated FullSync corrects it. No dead-letter table, no compensating mutation, no user surface.

## 2. WRONG / UNDERSPECIFIED — by doc section

**§3 (Seam).** `SearchResults` (3-domain) drops half the result surface (E). `AuthChallenge` (3.3) is too thin for RFC-8628 device-code: omits poll `interval`, can't express `slow_down`/`expired_token`/`access_denied`/cancel. `TrackPage`/`StreamTracksAsync` is the *only* paged read; `GetLikedSongsAsync`/`GetAlbumsAsync` return whole `IReadOnlyList<>` (`ICatalogSource.cs:36`) — a 50k list is the exact LOH cliff §10 claims to cure. Seam `Track.Artists` is a list but hot `TrackRow.ArtistSlot` is a single int — multi-artist sort/links can't be served from the hot row. Seam needs an entitlement signal (none exists) and `Resolve(uri)`.

**§4.1 (Schema).** Misattributed "v21 verbatim" — the proven schema is **v24** (`MetadataDatabase.cs:93`, ~30 tables); the outbox is **v22** append-only `id AUTOINCREMENT` with **no `idem_key`, no `(kind,uri)` uniqueness** (`EnqueueOutboxAsync` is a plain INSERT, `:2911`) — so the "coalesced end-state + idempotency key" the whole §8 rests on is *unbuilt net-new design dressed as a proven lift*, and `set_id TEXT PRIMARY KEY` does hold the `{set}:{itemType}` composite (the one dropped false-positive — clarify, not a gap). The cache key omits `(country, catalogue)` and `(locale)`: every POST stamps Country+Catalogue (`ExtendedMetadataClient.cs:386`); real tables key `localized_entities(uri, locale)` and `localized_extension_cache(uri, locale, ext)` (`:1249`). So §4.7's headline "a track in three playlists hydrates once, keyed by URI" is **incorrect** across a market/tier/locale change (travel/VPN/premium→free) — serves stale availability/relinking; §6.1 already parses market/tier so the invalidation trigger exists, unwired. Missing tables the seam already exposes: `play_history`, `color_cache` (→`Palette`, `Models.cs:116` — expensive extraction, never cached), `episode_progress`, `podcast_episodes`, `audio_keys`, rootlist/folders. `outbox` drains `ORDER BY created_at` (integer seconds, `:2940`) with **no `id` tiebreak** — two ops on one entity within a second drain in arbitrary order → wrong terminal state (the doc's own `logical_ts` would fix this but is unimplemented).

**§4.2 (Hot SoA).** Designed for `TrackRow` only. `Artist`+`ArtistExtras` is ~20 scalars plus nine nested list facets (`Models.cs:28`) — not POD-SoA-able; Album/Artist/Show have no gid→handle/eviction analog. State the honest position: non-track entities are on-demand-minted blobs.

**§4.6 vs §4.7 (mutually inconsistent).** A metadata sort/filter "completion fetch" must hold the entire 50k+ set in the arena simultaneously, directly contradicting §4.6 LRU eviction (you can't evict mid-sort). Neither section gives a byte budget, `OpenAddrMap` load-factor, or rehash story. Eviction must also drop the search-index entry (E) — unmentioned. The `SimpleSubject.OnNext` `ToArray()` per publish × 10k version bumps during full sync is the "bounded Gen0" the doc never numbers.

**§4.7 (Read pipeline).** Three wrong claims: (1) batch size is **not** "bounded only by POST-body limit — hundreds to low-thousands" — the real cap is a hard `HttpChunkSize = 500` with chunk-and-merge above it (`ExtendedMetadataClient.cs:311`). (2) The uniform two-phase shell→hydrate model breaks for GraphQL surfaces (discography/album/show return name/artist/duration **inline** via offset/limit — no skeleton step, no completion fetch needed) and for mixed/pseudo-URI library sets (playlist URIs have no bulk endpoint → per-item resolve; ylpin pseudo-URIs need placeholder `entities` rows — `SpotifyLibraryService.cs:1079`). (3) "The dealer subscription *is* the revalidation" rests on an unstated correctness gate: ops apply without a `/diff` round-trip **only if parent-revision equality holds** (`PlaylistCacheService.cs:1284`, `RevisionsEqual`); apply without it and you corrupt the list — and for parser-less sets (`hm://collection/ylpin` arrives `Items=[]`) it forces a full re-sync fallback the doc must keep. Track relinking is absent: seam `Availability` is binary, but proto `Track` carries `Alternative` (relinked gids), `Restriction`, and `PlayabilityRestriction` reasons (`EXPLICIT_CONTENT`/`AGE_RESTRICTED`/`NOT_IN_CATALOGUE`) — §7.4 checks `complete` by the *original* fileId while a relinked track plays the *alternative's* fileId, so the offline gate and the play resolve different entities. Entity set incomplete: no audiobook/chapter; seam `Album` lacks `Disc`/`Copyright`/`Label` (proto carries them, album page renders disc grouping); seam `Show` has a flat `Episodes?` with no `ConsumptionOrder`/paging; discography needs release-type buckets (Albums/Singles&EPs/Compilations).

**§5.3 (Federation table).** Outbox PK `(source, entity_uri)` can't encode per-set membership — one URI legally lives in multiple sets; "unpin but keep liked" needs `(source, entity_uri, set)` (`RemoveFromSpotifyLibraryAsync(uri, itemType)` exists *because* of this — `library-and-sync.md:211`). `CollectionKind` has 5 of 10 real values (`IMetadataDatabase.cs:758`); even the 5 modeled depend on `DetermineSetFromUri`'s `hm://collection/<set>/...` path parse (`LibraryChangeManager.cs:285`) the doc never specifies, so they can misroute. Pin/unpin is online-only with rollback (deliberately *not* outbox — `library-and-sync.md:404`); §5.3's `SetReplay` implies it's offline-queued — an offline pin is lost with a toast.

**§6.1 (Connection actor).** Beyond C/D/G/I/L above: `ProductInfo 0x50` is undermodeled as a static one-time mechanic — the live re-fold *exists* (`Session.cs:1180`, re-folds every push to `UserData`), but the *store* has no entitlement column and no UI-edge version bump, so a Free↔Premium / explicit-filter change never reaches the UI reactively. Two hardcoded server-allowlisted version strings (client-token + handshake `SpotifyClientIdentity.HandshakeBuildVersion`) with no maintenance surface. Rotated `ReusableAuthCredentials` aren't persisted on reconnect (`ReconnectApAsync` calls `SetStoredCredentials` but **not** `SaveCredentialsAsync` — `Session.cs:1406`), so a rotated blob is dropped and next launch uses the stale on-disk one. No 401-invalidate-and-retry, no 429 `Retry-After` backoff — *every* `Unauthorized`/`TooManyRequests` just throws (~20 sites in `SpClient.cs`); the §4.7 completion-fetch fan-out of thousands of URIs will hit 429 with zero defense. The dealer WS ping (30s/3s) is a *separate* heartbeat from the AP `Ping/Pong/PongAck` §6.1 conflates.

**§7 (Playback).** Premium-gating is entirely absent (Free = shuffle-only/restricted-on-demand, no seek, ads, bitrate cap, market-greyed) — §7.4 covers only offline playability; §7.2's reducer takes no entitlement input. Resolve is *not* "RequestAsync → cdnUrl+audioKey+fileId" — it's relink + quality-ladder + parallel fan-out, requesting the AudioKey against the **relinked** `effectiveTrackId` (`TrackResolver.cs:700,831`). Gapless/crossfade/prefetch staging is entirely absent (`PrepareNextAsync`, `CrossfadeProcessor`, AudioKey-first socket ordering — `playback.md`) — a product regression. §7.3/7.4 "cached-Spotify in-process" is false: `.enc` is AES-128-CTR needing the key, and premium keys need PlayPlay's out-of-process `Spotify.dll` — the corrupt-cache OggS self-heal means `complete` can still hand the engine an undecodable file. Position/seek model is wrong-cadence (real `state_update` ~5 s + 1 Hz UI interpolation; doc inverts to reducer-synthesized 1 Hz), models no buffering/rebuffer/`UrlAwareStream` ~1h mid-stream URL expiry, no typed `error`/`track_finished` IPC events. Autoplay is misplaced inside *pure* QueueCore (§7.1) — it's network I/O with an in-flight grace wait (`_pendingAutoplayTask`); the ladder must terminate at `EndOfContext` and surface an async request. Video (`track_player=video`) is a distinct PlayReady/`MediaPlayerElement` path, not the audio reducer. Active-device duality: when remote, queue/now-playing come **only** from cluster `PlayerState.ProvidedTrack`/`RawNextQueue` (`connect-state.md:83`), but §7.2 also declares `QueueCore` the authority — two authorities, no reconciliation; cluster `Restrictions` (grey controls) surface nowhere. Volume domain mismatch never converted: seam `double 0..1` vs `int 0–100` (`Playback.cs:53`) vs wire `0–65535`, arriving as a *MESSAGE* triggering a re-PutState echo.

**§8 (Sync).** The single coalesced-outbox abstraction is fundamentally incompatible with cursor-based playlist adds (`PlaylistAddTracksHandler.cs`: "MUST NOT dedupe," chunks at 500 URIs/Op, persists a `ProgressOffset` cursor to **resume not replay**) — needs two outbox disciplines. `playlist_items` keyed `(playlist_uri, position)` has **no `item_id` column** for the stable per-item identity REM-by-key and reorder require. Only `PlaylistAddTracksHandler` exists — no remove/reorder handler, no rebase (§8.5's "rebase by track-id" is unbuilt). `snapshot_id TEXT` is the wrong primitive: the model is base62 `bytes revision` + ordered `Op` deltas, and `multiple_heads` — the literal server "your edit forked" signal the §8.4 conflict policy must consume — is unmodeled. Cover-image edits are a server-minted two-step upload (`image-upload.spotify.com/v4`, JPEG ≤256 KB → opaque token → register) — **not offline-replayable**; the outbox silently assumes all writes replay. Cross-version outbox survival is unspecified and the real code *guarantees loss*: v22 `DROP TABLE library_outbox`, `OutboxProcessor.cs:75` drops unknown `op_kind` (downgrade) — violating "writes are durable intents" exactly at migration. §8.3's delta-too-stale fallback is `FullSyncAsync` → `ClearSpotifyLibraryAsync` (`library-and-sync.md:326`) which deletes the collection *including optimistic Pending rows* before the outbox drains — the doc never carves out preservation. ## 4. NET ASSESSMENT

**The architecture is sound and does not need a structural rethink — but it is under-scoped by roughly a third, and two of its load-bearing *claims* are factually wrong in ways that will bite in month 3.** The core bets all survive: the store-as-source-of-truth spine, source federation, the populator model, the optimistic-outbox write path, the pure QueueCore, and the one-separate-audio-process decision are all the right shapes and accommodate every gap above as *additions*, not reversals. Nothing here forces abandoning the design.

Three caveats temper that verdict. **First, the `entities`-as-opaque-blob decision (§4.1) is the one choice that is locally structural** — it silently breaks offline search, makes every metadata sort/filter an all-set in-memory completion fetch that contradicts §4.6 eviction, and must be reversed to projected NOCASE-indexed columns (the proven WaveeMusic shape). That is a small schema change but it ripples into §4.6/§4.7, so make it first. **Second, the doc systematically misrepresents net-new design as "v21/v24 verbatim lift"** — the coalesced+idempotent outbox is *unbuilt* (the real code is append-only, cursor-based, lossy across migration), the two-phase hydrate is *wrong* for GraphQL surfaces, and the cache "hydrates once by URI" is *wrong* across market/locale. These aren't missing features; they're confident claims the ground truth contradicts, and they are exactly the kind of thing that passes review and fails in production. **Third, the "two primitives" minimalism in §6.1 is genuinely too small** — it cannot express the dealer REQUEST→reply contract, the off-socket HTTPS credentials (client-token/login5), or the gabo side-effect stream. The fix is additive (a third reply primitive + an HTTPS lane + an emitter actor), not a redesign, but the doc's elegance here is buying a correctness bug.

With the §3 additions (download manager, search index, play-history, Home pipeline, folders, gabo, client-token, per-account partitioning, entitlement signal) the design is shippable and genuinely offline-capable. The roadmap effort estimate (~5–7 months solo) is **optimistic by ~2–3 months** given these are nine real subsystems, not polish — and offline download + search + Home are not deferrable past v1 for an "offline-capable Spotify client."
