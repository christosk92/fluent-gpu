# Wavee Spotify Connect + Playback: Architecture Analysis & fluent-gpu Mapping

*Author's note for the reader: you wrote both codebases. This document does not re-explain Spotify or your engine; it (a) inventories what WaveeMusic actually does on the wire and in memory, (b) is blunt about which parts are correctness/allocation traps you must not port, and (c) lands a concrete fluent-gpu design that reaches **everything up to the sound-producing host**, with that host deferred behind one clean seam. All claims cite `file:line`; where a map and the code disagreed I re-read the source — those corrections are called out inline. This revision folds in an adversarial review: the readiness of the audio-key path was overstated, the live dealer is not yet composed into the running app, and three control-plane pillars (token refresh, a persistent AP channel, outbound remote control) plus an optimistic-reconciliation policy were missing. They are designed below.*

---

## 1. Executive summary

**Goal.** Bring real Spotify Connect + playback to Wavee-on-fluent-gpu: register this device in the Connect cloud, receive/dispatch/ack remote commands, project a bidirectional now-playing state, resolve tracks (full-metadata file-select + **audio-key fetch** + **storage-resolve/CDN locate**), wire the UI — and stop at a single, process-agnostic `IAudioHost` seam, behind which a **silent/stub host** stands in for the deferred sound producer so the whole pipeline runs end-to-end, headlessly, today.

**Current state — WaveeMusic (the reference, `C:\WAVEE\WaveeMusic`).** Wire-correct and UX-mature, but built on the deliberate inverse of your engine's substrate: System.Reactive everywhere, per-tick `record with` + per-publish protobuf build/`Clone`/`Guid`, god-classes (`PlaybackStateManager` 1365 LOC + `PlaybackStateHelpers` 1430; `PlaybackOrchestrator` 2778; `DealerClient` ~980; `PlaybackQueue` 1316), WinUI/MVVM/DI coupling, a reflection-tail reply serializer, and an out-of-process x64 `AudioHost` whose *true* reason to exist is loading the x86_64 `Spotify.dll` for PlayPlay key derivation — crash/GC isolation are secondary (`Wavee.AudioHost/README.md:7-14`). WaveeMusic's own `CLAUDE.md:247` confesses a GC odyssey chasing 1–2 s Gen2 pauses rooted in "gross transient allocation (protobuf / JSON / image-decode landing on the LOH)". That is precisely the failure mode your engine exists to avoid — so this is a **behavior port, not a class port**. Crucially, WaveeMusic *does* keep a long-lived encrypted AP socket (the basis of its `0x0c` audio-key path); fluent-gpu does not yet (see below) — that is the single biggest in-scope build.

**Current state — fluent-gpu Wavee (`C:\WAVEE\fluent-gpu\app`).** The *shapes are already right and a few pieces are live — but less of the live wiring is in the running app than a first read suggests*:
- The dealer wss firehose is real (`SpotifyLive/LiveDealerTransport.cs`) with a tested frame parser that already classifies REQUEST frames (`DealerFrameParser.Parse`, consumed at `LiveDealerTransport.cs:96`) and **already gunzips gzip frames in-parser** (`DealerFrameParser.cs:42,46`). **Correction (vs. the maps and the prior draft):** it is instantiated **only** inside the transient `--spotify-sync` CLI probe (`SpotifyLibrarySync.cs:60-64`), which listens for 20 s (`:66-67`) and then the caller `Environment.Exit`s (`Program.cs:124-125`). The running UI app opens **no** dealer.
- `SimpleSubject<T>` *is* WaveeMusic's `SafeSubject` done zero-dependency (cached copy-on-write snapshot, zero per-publish alloc, last-value-on-subscribe).
- `PlaybackBridge` *is* the WaveeMusic `PlaybackStateService`/INPC UI projection, done correctly — one Core-`IObservable`→engine-`Signal` boundary, every callback marshalled onto the UI thread via `post(...)` (`PlaybackBridge.cs:64-76`), optimistic-write-then-reconcile, with the hot position bound as a compositor `FloatSignal` that never re-renders (`SeekBar.cs:119-136`). It is wired in *both* `CreateFake` and `CreateReal` — but in both cases over the in-memory `FakePlaybackProvider`.
- The reducer→event-log→projection model already exists (`Backend/Playback.cs`), and the comment at `Backend/Playback.cs:147` literally names the missing piece: *"The seam IPlaybackState is one such projection."* **Correction:** that model is an **island** — `PlaybackReducer`/`QueueCore` are exercised *only* by `BackendSelfTest` (`Scaffold.cs:27,112`); the running app uses `FakePlaybackProvider` for `IPlaybackState`/`IPlaybackPlayer` in *both* `CreateFake` (`Services.cs:57`) and `CreateReal` (`Services.cs:109`). Hosting the projection on the reducer therefore requires *structural* work in the composition root, not just a new class.

The gaps are bounded but more numerous than "four no-ops":
- `ITransport.Publish` is a no-op (`Transport.cs:41`, `LiveDealerTransport.cs:59`); `ITransport.Reply` hard-codes `"success":true` with no result code, no request tracking, **and concatenates `requestId` into JSON unescaped** (`LiveDealerTransport.cs:56-57`); the receive loop forwards only `Message` frames and **drops parsed REQUEST frames** (`LiveDealerTransport.cs:98`); `WireEvent` carries no headers, so the `Spotify-Connection-Id` is never surfaced (`Transport.cs:17`, `WireEvent(string Topic, byte[] Payload)`).
- There is **no** reducer→`IPlaybackState` bridge wired into the app (the only `IPlaybackState` the app uses is `FakePlaybackProvider`), and `IAudioEngine`/`StubCrypto`/`StubAudioEngine` are passthrough placeholders (`Backend/Playback.cs:108-139`).
- **No live-session lifecycle owner exists.** `CreateReal` wires `StubTransport` (`Services.cs:123`) + `FakePlaybackProvider` (`Services.cs:109`); the live dealer lives only in the CLI probe; the transport's access-token provider is a captured **constant** (`SpotifyLibrarySync.cs:60`, `_ => Task.FromResult(live.AccessToken)`) even though `LiveDealerTransport` re-invokes it on every reconnect (`:67`) and login5 tokens **expire** (`Login5Client.cs:29,45`).
- **No persistent AP channel exists** for audio-key fetch: login disposes its socket immediately after APWelcome (`SpotifyLiveLogin.cs:83-84`, `using var tcp`), and `SpotifyConnection.HandshakeAndLoginAsync` returns only `SpotifyWelcome` (`SpotifyConnection.cs:29,156-158`) — never the channel. `Backend/Spotify/ApChannel.cs` is the *simplified, protobuf-free test channel* (its own header says so, `ApChannel.cs:9-11`); there is no `0x0c`/`0x0d` exchange anywhere (the only `0x0c` in the tree is RSA-modulus bytes, `Handshake.cs:72,75,76`).
- **No file-id source for the resolver:** the domain `Track` carries none (`Models.cs:76-89`) and the live metadata path is **Lean-only**, which deliberately skips `file[]` (`ExtendedMetadataSource.cs:27`; `lean_metadata.proto:5-6,44-51`).

**Headline recommendation.** Build the unbuilt pillars in dependency order, behind the seams that already exist, decomposing WaveeMusic's god-classes into single-responsibility collaborators and re-hosting its hard-won correctness on POD/slab paths:
0. **Live-session lifecycle + token refresh** (the prerequisite the prior draft omitted) — an in-app owner that holds the reusable credential + a `Login5Client`, feeds the transport a *re-minting* access-token provider, stands up the persistent transport/AP channel, and swaps the `FakePlaybackProvider`-backed `Services` facets for the real ones on connect.
1. **Transport widening** — header-carrying frames (for `connection_id` + REQUEST `message_ident` — **not** for gzip, which is already handled) + `Requests(prefix)` + typed `Reply(id, result)` (escaped) + real `Publish` + a `ConnectionId` surface.
2. **`ConnectService`** (the decomposed `DealerClient`) — `ConnectionId` signal + `ConnectCommandRouter` + `ClusterIngest` + a 10 s reply backstop, all on the single `Events("hm://")` firehose.
3. **`NowPlayingProjection` + `DeviceStateProjection`** over a **PlaybackState POD slab** + **ConnectDevice roster** — the central reducer→`IPlaybackState`/PutState bridge — plus an explicit **optimistic-reconciliation policy** (command-in-flight gating, stale-cluster suppression, echo correlation, proxy precedence).
4. **`PlaybackController`** with a **local-vs-remote-forward branch** (us-as-target → local host; another device active → outbound remote command), the missing twin of WaveeMusic's `ConnectCommandClient`.
5. **`ITrackResolver` + a persistent AP-channel `IAudioKeySource` + storage-resolve** → an `AudioStreamHandle`, handed to a reshaped **`IAudioHost`** whose default impl in this scope is a **`SilentAudioHost`**.
6. **A Connect test harness with a real GC-delta gate** — `BackendSelfTest` has none today (`Scaffold.cs:32-141`); the engine's alloc tripwire (`FluentGpu.VerticalSlice`) does not touch this backend, so the gate must be built.

**Explicit scope line (per 2026-06-27).**
> **IN SCOPE, designed in full:** the Connect control plane (device registration / PutState announce, remote command receive → dispatch → ack, **outbound remote control** when another device is active), the bidirectional playback-state projection (cluster ⇄ now-playing `IPlaybackState` ⇄ PutState) **with its optimistic-reconciliation policy**, the queue/controller actions + transfer (in **and** out), the live-session lifecycle + **dealer token refresh**, track resolution (**incl. full-metadata file-select**), **audio-KEY FETCH** (over a new **persistent AP channel**), **storage-resolve/CDN locate**, the UI wiring, and a **silent/stub `IAudioHost`** behind the seam.
> **DEFERRED behind `IAudioHost` (future, not designed here):** AES-128-CTR decrypt-to-PCM, Ogg/Vorbis decode, mixer/DSP, audio output, the out-of-process/named-pipe host apparatus, and the PlayPlay/x86_64 native **key-derivation fallback**. The seam sits **after** storage-resolve + key-fetch and **before** decrypt/decode/output.

## 2. WaveeMusic architecture, end to end

The control plane is one wss "dealer" firehose plus a handful of spclient HTTP routes, **plus a long-lived encrypted AP socket** (Mercury/audio-key/packets). Data flows: **dealer transport + AP socket → connect-state device registration → command receive/dispatch/ack → playback-state projection (cluster ⇄ local ⇄ PutState) → queue/orchestration → track resolution → storage-resolve + key-fetch → [AUDIO HOST]**.

### 2.1 Dealer transport (the wss firehose)
**Files:** `Connect/DealerClient.cs` (~980 LOC god-object), `Connect/Connection/DealerConnection.cs` (ClientWebSocket + System.IO.Pipelines), `Connect/Protocol/MessageParser.cs`, `Connect/SafeSubject.cs`, `Connect/HeartbeatManager.cs`, `Connect/ReconnectionManager.cs`, `Core/Session/ApResolver.cs`.

**Data flow.** `ConnectAsync` → `GetAccessTokenAsync` → `ApResolver.ResolveDealerAsync` (HTTP, fallback `dealer.spotify.com:443`) → `wss://{dealer}/?access_token=…` (`DealerClient.cs:169-246`). `DealerConnection` runs a producer/consumer Pipe (pause 1 MB / resume 512 KB, `DealerConnection.cs:55-59`); one WS message per flush; single-segment is zero-copy, multi-segment is `ToArray` (`DealerConnection.cs:385-408`). `MessageParser` classifies four frame types (ping/pong/message/request), auto-PONGs server pings (cached static bytes, `DealerClient.cs:55-60`), and reads the **`Spotify-Connection-Id` HEADER** (`DealerClient.cs:806`, `headers.TryGetValue`) — *gated by* the `hm://pusher/v1/connections/<id>` URI prefix, not derived from the URI (`DealerClient.cs:486-489, 796-816`) → `_connectionId.OnNext`: the required gate for device announce. *(This header-vs-URI distinction is the justification for the header-surfacing work in §5.)* REQUEST payloads are gzip-inflated (`MessageParser.cs:222-248`). Decoded frames fan out across two unbounded `AsyncWorker<T>` channels (message/request) onto `SafeSubject<T>` — **subscribers run inline on that one worker thread, no `ObserveOn`** (`PlaybackStateManager.cs:328-333`).

**Key types.** `DealerClient`, `IDealerConnection`/`DealerConnection`, `SafeSubject<T>`, `HeartbeatManager`, `ReconnectionManager`, `AsyncWorker<T>`, `DealerMessage`/`DealerRequest` records, `MessageType`/`RequestResult` enums, `PendingRequest`.

### 2.2 Connect-state device registration (PutState announce)
**Files:** `Connect/DeviceStateManager.cs` (the *only* writer of `PUT /connect-state/v1/devices/{id}`), `Connect/ConnectStateHelpers.cs` (DeviceInfo/Capabilities/PrivateDeviceInfo), `Core/Http/SpClient.cs:575` (`PutConnectStateAsync`).

**Data flow.** On the first `connection_id` (null-filtered subscription, `DeviceStateManager.cs:90-92`) → `UpdateStateAsync(PutStateReason.NewConnection)`. Player_state resolves *pending > live snapshot > empty* (`DeviceStateManager.cs:248-271`). The `PutStateRequest` (`MemberType.ConnectState`, `Device{DeviceInfo,PlayerState,PrivateDeviceInfo}`, `PutStateReason`, `IsActive`, `ClientSideTimestamp`, `MessageId++`) is gzip-protobuf'd and PUT with `X-Spotify-Connection-Id` + `X-Transfer-Encoding:gzip` (`SpClient.cs:590-664`); the returned **Cluster is re-injected as a synthetic dealer message** `hm://connect-state/v1/put-state-response` (`X-Wavee-Echo:self`) so the announce-response and push-cluster paths unify (`DeviceStateManager.cs:309-330`). Capabilities are matched to desktop 1:1 — `NeedsFullPlayerState=true`, `VolumeSteps=64`, 14 `supported_types`, volume range `0..65535` (`ConnectStateHelpers.cs:127-167`).

### 2.3 Command receive → dispatch → ack
**Files:** `Connect/Commands/ConnectCommandHandler.cs`, `Connect/Commands/ConnectCommand.cs` (`Parse` factory), `Connect/ConnectCommandClient.cs` (**outbound — remote control of *other* devices**), `Audio/PlaybackOrchestrator.cs:2212` (`SubscribeToRemoteCommands` → 9 engine verbs), `Session.cs:327-340` (transfer → `SetActiveAsync`).

**Data flow.** A REQUEST is tracked with a timeout CTS; no-observer → immediate `DeviceNotFound` reply (`DealerClient.cs:500-524`). `ConnectCommandHandler` filters `hm://connect-state/v1/`, `ConnectCommand.Parse` reads the nested `command.endpoint` into typed records (play/pause/resume/seek/skip_*/set_*/transfer/queue-ops, `ConnectCommand.cs:47-124`), a bounded worker (cap 100) dispatches per-type `SafeSubject`s, then **acks `Success` right after dispatch** within the 10 s SLA (`ConnectCommandHandler.cs:229-305`). Reply frame = `{type:reply,key,payload:{success}}`, `success = result==Success` (`DealerClient.cs:254-321`); a 5 s timer replies `UpstreamError` for anything > 10 s (`DealerClient.cs:822-887`). **Outbound** (`ConnectCommandClient`) is the inverse: when WaveeMusic is the *controller* of another active device, it SENDS player commands to that device — the path the fluent-gpu mapping (§5) was missing.

### 2.4 Playback-state projection (cluster ⇄ local ⇄ PutState)
**Files:** `Connect/PlaybackStateManager.cs` (god-class, 1365 LOC), `Connect/PlaybackStateHelpers.cs` (pure protobuf↔domain, 1430 LOC; `ToPlayerState` ~310), `Connect/PlaybackState.cs` (unified immutable record), `Wavee.UI.WinUI/.../PlaybackStateService.cs` (INPC bridge, 2010 LOC).

**Data flow.** *Inbound:* `Messages.Where(hm://…/cluster* | put-state-response*)` → gunzip → `ClusterToPlaybackState` merges `PlayerState` onto the previous unified `PlaybackState` (active-device lookup, queue extraction, `position_as_of_timestamp` vs `position`, no-active-device Playing→Paused clamp), `DetectChanges` sets `[Flags]`, then mutate `_currentState` + `_stateSubject.OnNext` under `_stateLock` (`PlaybackStateManager.cs:469-632`). *Outbound:* local engine `StateChanges` → `LocalToPlaybackState` (mints stable per-context `SessionId` / per-track `PlaybackId`) → critical changes flush immediately, others debounce 750 ms, dedup vs `_lastPublishKey`, `request.Clone()` deep-clone, bounded (cap 10) drop-on-full publish worker → `PutConnectStateAsync` (`PlaybackStateManager.cs:719-1092`). The device roster is propagated independent of playback suppression so the picker stays live in proxy-only / we-are-active modes. **Optimistic reconciliation** (the part the prior draft left as prose): a local command writes UI state immediately, then a *stale* cluster push must not revert it — WaveeMusic leans on the self-echo + the publish-key dedup + the active-device check; this is its highest-risk area (§3.1 item 6, §5).

**Key types.** `PlaybackState` (unified record), `LocalPlaybackState` (~60-prop engine DTO), `StateChanges [Flags]`, `StateSource`, `PlaybackStatus`, `ProvidedTrack`/`Cluster`/`ClusterUpdate`/`PutStateRequest` (protobuf).

### 2.5 Queue / orchestration → track resolution → storage-resolve + key-fetch
**Files:** `Audio/PlaybackOrchestrator.cs` (2778 LOC), `Audio/Queue/PlaybackQueue.cs` (3 buckets, dual cursor, FNV revision), `Audio/ContextResolver.cs`, `Audio/TrackResolver.cs`, `Audio/TrackResolution.cs`, `Core/Audio/AudioKeyManager.cs`, `Core/Audio/HeadFileClient.cs`, `Audio/NormalizationData.cs`.

**Data flow.** `PlayAsync` resolves a context to a flat enriched tracklist (pagination ≤10 pages, retry/cooldown, 3-tier metadata, autoplay/radio-apollo continuation, `ContextResolver.cs:82-267`); the **3-bucket queue** (Play-Next / context / Add-to-Queue) drains user→context→post-context with `context_uri` frozen per-track at insertion (`PlaybackQueue.cs:264-384`). The standout pattern is **deferred resolution / head-start**: `TrackResolver.ResolveWithHeadAsync` selects the OGG Vorbis file from the **full** `Track.file[]` → `FileId`, then fires three parallel tasks and **awaits only the ~128 KB head** — `GetHeadDataAsync` (unauth CDN, `HeadFileClient.cs:81`), `AudioKeyManager.RequestAudioKeyAsync`, `GetCdnUrlAsync` — returning a `TrackResolution` whose key/CDN/size complete in the background (`TrackResolver.cs:404-487`). **Audio-key fetch:** in-mem → SQLite → **AP socket `RequestKey` raw packet `0x0c`/`0x0d`** (2.5 s/attempt, 5 retries, AP-reconnect after 2 timeouts) → PlayPlay HTTP fallback with a session-disable latch (`AudioKeyManager.cs:124-314`). **Note:** `0x0c` is a *raw AP packet command* over the long-lived encrypted socket — it is **not** Mercury (Mercury is the separate `hm://` request/reply framing). **Storage-resolve:** 30-min cache or `ResolveAudioStorageAsync(fileId)` → `StorageResolveResponse.Cdnurl[0]` (`TrackResolver.cs:97-112`). Normalization (ReplayGain) is parsed from head offset 144 (`NormalizationData.cs:25-62`); only track gain is forwarded. The resolved track is then handed to the audio host.

### 2.6 [AUDIO HOST: decrypt / decode / mix / output] — **DEFERRED (context only)**
**Files (summarized, not ported):** `Wavee.AudioHost/*`, `Wavee.AudioIpc/AudioPipelineProxy.cs`, `Wavee.AudioIpc/AudioProcessManager.cs`, `Wavee.Playback.Contracts/IpcMessages.cs`.

For context only: a separate **x64 process** receives a JSON-over-named-pipe `play_track` command (with base64 head for instant start), opens a ranged CDN stream (`ProgressiveDownloader`), applies **AES-128-CTR with the fetched key** (`AudioDecryptStream.cs:13-19`), strips the 167-byte (`0xA7`) Spotify header and **Ogg/Vorbis-decodes via NVorbis** to 16-bit PCM (`VorbisDecoder.cs`), runs a DSP chain (normalize/EQ/compressor/limiter), and outputs via **PortAudio→WASAPI shared mode** through a lock-free single-reader ring with a zero-alloc native callback (`PortAudioSink.cs:261-325, 1239-1374`). PCM never crosses the pipe; only compact `state_update`/`track_finished` do. The host self-justifies on `README.md:7-14`: it is x64 **because** PlayPlay must `LoadLibrary` the x86_64 `Spotify.dll`. **Everything in this subsection is deferred behind `IAudioHost` in the fluent-gpu design (§6); the host internals are not designed here.**

### 3.3 The dealer hot-path is *not* free (the allocation the prior draft waved through)
§4.4 below classifies the whole control plane as "GC-tolerant," but be honest about *what* it tolerates: the live receive path allocates **per frame**. `DealerFrameParser` does a Base64 decode (`DealerFrameParser.cs:79`, `GetBytesFromBase64`) and, for gzip frames, a `MemoryStream→CopyTo→ToArray` gunzip (`DealerFrameParser.cs:84-90`); the receive loop mints a new `WireEvent` + `uri` string per message (`LiveDealerTransport.cs:98`). Under a busy Connect session — a remote device pushing ~1 Hz position/seek clusters — that is **sustained Gen0 churn at the event edge**. It is *not* in frame phases 6–13, so it does not break the engine tripwire; but it is also not "zero," and nothing currently bounds it. The design must either (a) pool the payload `byte[]` and reuse `WireEvent`, avoiding the per-frame Base64/gunzip intermediates where the frame isn't needed, or (b) *explicitly accept* bounded Gen0 here with a stated rationale (human-1 Hz cadence, Workstation+Concurrent GC). Decision in §8.

## 4. fluent-gpu target: the idiom and the constraints that bind this design

Sources: `MAP 6` (backend seams), `MAP 8` (consumption/signals), `MAP 9` (engine constraints), `MAP 7` (composition root), and the verified seam files in §1.

### 4.1 The two-tier seam (the boundary the port must satisfy)
- **`Wavee.Core`** — portable, framework-neutral POD, **zero package deps** (no System.Reactive). App-facing contracts the UI consumes: `IPlaybackPlayer`, `IPlaybackState`, `IConnectDevices` (`Wavee.Core/Playback/Playback.cs:8-61`), `SimpleSubject<T>`, domain models. **These must not move** — `PlaybackBridge` and every UI consumer bind to them.
- **`Wavee.Backend`** — the engine-internal impl (namespace `Wavee.Backend`): the five engines (Store/Resource/Mutation/Session/Transport), `ITransport`, `IAudioEngine`, `IPlaybackProjection`, `QueueCore`/`PlaybackReducer`, bridged to `Wavee.Core` by thin `Seam.cs` adapters.
- **`Wavee.SpotifyLive`** — the real wire (`LiveDealerTransport`, AP/Shannon/Mercury/login5).
- **`Wavee` (App/Features)** — the FluentGpu UI + `PlaybackBridge`/`Services.cs` composition root.

Net: **keep the `Wavee.Core` contracts unchanged; implement the `Wavee.Backend` path; bridge with `Seam.cs`-style adapters.** A real now-playing `IPlaybackState` is *just another `IPlaybackProjection`* (`Backend/Playback.cs:147`). **But:** today `CreateReal` does not wire that path — it wires `StubTransport` (`Services.cs:123`) + `FakePlaybackProvider` (`Services.cs:109`), and the reducer is only reachable via `BackendSelfTest` (`Scaffold.cs:27,112`). Replacing the fake with the reducer/controller/projection path in the composition root is itself a deliverable (§5.2(a), §7 Stage 0).

### 4.2 The reducer → event-log → projection model (already the right shape)
`PlaybackReducer` (`Backend/Playback.cs:148`) turns intents into a pure `QueueCore` drive + an `IAudioEngine` drive + a `PlaybackEvent` log (`EvKind {Started,Paused,Resumed,TrackChanged,Ended}`, `:142-143`) fanned to independent `IPlaybackProjection`s. Today the only projection is `HistoryProjection` (`:191`). **Corrections to the maps:** `QueueCore` is *single-cursor* (`_cursor`, `:18`) with a **single** front-drained user queue (`_userQueue`, `:17`; `Next()` does `_userQueue.RemoveAt(0)` without moving `_cursor`, `:44`) — so the "dual cursor, 3-bucket" framing in the code comment (`:10`) overstates it: there is *one* context cursor + *one* front-drained user queue, and **no per-entry `context_uri` freeze** (the cited `:41-49` is the `Next()` ladder, not an insertion site). And `Snapshot()` currently emits **only** `NowPlaying + UserQueue` (`:98-105`); it omits the context next-up window — a queue panel beyond the user queue would render empty until both the buckets and `Snapshot` are widened.

### 4.3 Signals-first + the slab idiom
Playback state is `Signal<T>`; a write *value-eq-gates then `MarkStale`s* exactly the render-effects that read it, drained once per frame in `ReactiveRuntime.Flush` (phase 3). Hot scalars (`PositionFrac`, `Volume`) are `FloatSignal` compositor binds — value→pixels with **no re-render** (`SeekBar.cs:119-136`). POD change tokens (`StoreChange`, `Resp`, `WireEvent`, `PlaybackEvent`) are `readonly record struct` precisely to avoid heap churn/boxing through `SimpleSubject`. The keyed-POD-slab precedent is `AnimValue`/`ScrollBind`: one mutable struct mutated under a gate, fired outside it — the model the new PlaybackState slab follows.

### 4.4 The binding constraints (and the honest scope of each)
- **Near-zero-alloc — but know *where*.** The hard tripwire (`GC.GetAllocatedBytesForCurrentThread()` delta == 0) binds **frame phases 6–13** (record/batch/submit/present) and is enforced by `FluentGpu.VerticalSlice` in the *engine* — **it does not touch the Wavee backend at all.** The Connect/projection/resolution code is the **event-driven app/data layer, *outside* those phases** — it is GC-*tolerant* (bounded Gen0 at the event edge, human-timescale cadence). So the PlaybackState slab is about not re-importing WaveeMusic's per-tick `record with`/`Clone`/`Guid` churn (matching the engine's POD-token bar), **not** about phase-6–13 zero-alloc — the projection never runs there. The dealer *parse* path additionally allocates per frame (§3.3) and is likewise unguarded. What *is* tripwire-bound is the SeekBar's `FloatSignal` bind. **Correction to the prior draft:** there is no existing alloc gate on this backend — `BackendSelfTest` is functional pass/fail with zero GC assertions (`Scaffold.cs:32-141`). A GC-delta assertion on the projection/ingest hot path must be *built* (§5.2(k), §7).
- **Position must be a coalesced Driven signal, never a per-frame write.** A naive per-frame `PositionMs` write would free-run the loop at display rate (the high-CPU-ambient-fps regression class the `Signal` `SubscriberCount` guardrail names). Use `SignalSourceKind.MediaClockMs` (`AnimValue.cs:113`) + `Cadence.DrivenBy(slot)` (`AnimClock.cs:17,80`): paused ⇒ +inf next-due ⇒ zero frames, event-woken by the ~1 Hz position write; the UI interpolates between ticks on the compositor — **the wall-clock anchor lives in `SeekBar` (`SeekBar.cs:63`, `_tickPosMs + (TickCount64 - _tickWallMs)`), not in any provider.** **Maturity caveat:** the min(next-due) gating is still a stub (`AnimScheduler.cs:43`) and `AdvanceTimeline` still samples the legacy `DrivenClockTable Func<float>` (`AnimScheduler.Timeline.cs:75`) — so the zero-frames-when-paused benefit is not fully live until the index-based `SignalSource` table is wired (sequencing decision, §8).
- **NativeAOT + trim (per-exe `PublishAot`).** No Reflection.Emit / `Expression.Compile` / dynamic proxies / assembly-scan / `BinaryFormatter` on any path. Replies/PutState/Connect protos use source-gen (`JsonSerializerContext` / `Wavee.Protocol` protobuf). DI is hand-wired (`Services.cs`, plain `new`). The Ogg decoder (deferred) must be pure-managed-AOT-safe or native `[LibraryImport]`. **Correction to the prior draft's "contra CLAUDE.md" caveat:** the app *does* have its own `Directory.Build.props` (`app/Directory.Build.props:3-7`) that deliberately does **not** inherit `src/Directory.Build.props` and instead *mirrors* its posture (net10, `LangVersion 14`, Workstation+Concurrent GC at `:15-17`); `PublishAot` is set only on the WinExe (`Wavee.csproj:8`); `TrimMode full` is **not set in the app at all**; and `GCSettings.SustainedLowLatency` is a runtime call in engine hosting, not a csproj property. So this is *consistent with* CLAUDE.md (which describes the `src/` tree), not contrary to it.
- **Threading / render-thread seam — and a NEW confinement domain.** Four strictly-confined domains: (1) **UI thread** (reconcile/layout/hooks/SceneStore, touches zero COM); (2) **render thread** (owns every `ComPtr`, records/submits/presents); (3) **worker pool** (pure decode over immutable inputs, crosses back only as POD); (4) **[deferred] audio-device thread** (WASAPI) — a *new* domain that must touch no Scene/`ComPtr`/SceneStore and cross back **only as a POD `AudioHostSignal`**. The control plane here lives *above* the render seam: it only writes `Signal`s the UI thread reconciles. The cross-thread rules it must honor now: every Core observable callback is `post(...)`-marshalled onto the UI thread before any `Signal` write (`PlaybackBridge.cs:64-76`); and the **dealer thread**, the **AP read-pump thread** (§5.3), and the host's `Ended`/position must all be serialized onto the controller before touching `QueueCore`/`PlaybackReducer` (which are not thread-safe — §5, §8).
- **Residency precedents (for the deferred host).** `ImageCache` (dedup → handle, Pending/Ready/Failed, Pin/Unpin ref-count, LRU byte-budget, Prefetch, +1-frame Pump) and `DecodeScheduler` (priority lanes, `clamp(cores-2,2..6)` workers, POD `Req`/`Done`, ArrayPool, backpressure/cancel) are the 1:1 templates for an `AudioCache`/`StreamScheduler` — with a *stricter pin*: the playing+next PCM ring must never be LRU-evicted mid-playback.

**(i) `PlaybackController` + the local-vs-remote branch** *(new)*. Owns `QueueCore` + `ITrackResolver` + `IAudioHost` + the projection fan-out, and is the single serialization point for dealer-thread commands, AP-pump signals, and host `Ended`/position. Its dispatch has two arms keyed on the active device:
```
if (slab.activeDeviceId == us)  → local: resolve → IAudioHost.Load+Play / Pause / Seek …
else                           → remote: ConnectCommandClient-equivalent → ITransport (spclient player command / dealer SEND)
```
This is the twin of WaveeMusic's `ConnectCommandClient` (us-as-controller) that the prior draft omitted. The route (spclient vs dealer) and the precise active-device rule are decisions (§8).

**(j) Optimistic-reconciliation policy** *(new — the §1 "optimistic-write-then-reconcile" claim made concrete)*. Four rules, all expressible on the slab (`lastLocalCmdWallMs`, `inFlightCmdSeq`):
1. **Command-in-flight window:** after a local command, stamp `lastLocalCmdWallMs` + bump `inFlightCmdSeq`; for a short window (or until our echo returns) a cluster push that *contradicts* the optimistic field is merged, not applied verbatim.
2. **Stale-cluster suppression:** drop/clamp a cluster whose `position_as_of_timestamp`/server clock predates the last local command for the same track.
3. **Echo correlation:** extend the `X-Wavee-Echo:self` trick (used for PutState in WaveeMusic, `DeviceStateManager.cs:309-330`) to command acks so our own round-trip isn't treated as a foreign change.
4. **Proxy precedence:** when `activeDeviceId != us`, **cluster wins** (we are a viewer); when `== us`, local wins modulo confirmed cluster fields. Settle the exact window + precedence in §8.

**(k) Connect test harness + alloc gate** *(new — it does not exist)*. `BackendSelfTest` is functional only (`Scaffold.cs:32-141`). Build a harness that scripts a dealer REQUEST → typed-ack → cluster-echo → projection round-trip, and assert `GC.GetAllocatedBytesForCurrentThread()` delta against a *stated budget* on the ingest/projection hot path (not 0 — bounded; §3.3, §8).

**(l) MediaClock Driven source** — bind position to `SignalSourceKind.MediaClockMs` + `Cadence.DrivenBy(slot)` so a paused playhead costs zero frames (sequencing caveat, §4.4 / §8).

> **Required refactor surfaced by the seam reshape.** Today `PlaybackReducer` *itself* drives `_audio` (`Backend/Playback.cs:151,172,181-182`) with a bare `Track`. Once `IAudioEngine.Load(Track)` becomes `IAudioHost.Load(in AudioStreamHandle)`, the **async resolve must move out of the reducer into the `PlaybackController`** (§5.2(i)); the reducer becomes pure queue+event-log; the controller does resolve→`Load`+`Play`, `Ended→Next()`, **and** the remote-forward branch — and is the single thread on which dealer/AP-pump/host signals are serialized (§8).

### 5.3 Persistent AP channel for audio-key fetch *(new subsection — the single biggest in-scope build)*

The audio-key path needs a **long-lived encrypted AP/Shannon channel**; the runtime keeps none. Login opens an AP socket transiently and disposes it immediately after APWelcome (`SpotifyLiveLogin.cs:83-84`, `using var tcp`), and `SpotifyConnection.HandshakeAndLoginAsync` returns only `SpotifyWelcome` (`SpotifyConnection.cs:29,156-158`) — never the channel. The real mechanics it *does* have are sound and tested: `DiffieHellman` + `ApSignature` verification + `ApKeyDerivation` + the Shannon `ApCodec` (`SpotifyConnection.cs:72-117`), and a generic encrypted send/receive (`ApChannel.cs:109-121`) — but `ApChannel.cs` is the *simplified, protobuf-free test channel* (`:9-11`), not the live path.

Design:
1. **Retain the channel.** Refactor `SpotifyConnection.HandshakeAndLoginAsync` to return an `ApConnection` (the live `IDuplexStream` + the negotiated `ApCodec`) **in addition to** `SpotifyWelcome`, instead of letting the socket fall out of the `using` (`SpotifyLiveLogin.cs:83`). The `LiveSession` owner (§5.2(a)) holds it.
2. **One read-pump.** Promote the post-login receive loop (`SpotifyConnection.cs:128-150`) to a long-lived pump that decodes each packet (`ReceivePacketAsync`, `:161-170`) and dispatches by `cmd` to registered waiters/handlers (audio-key, Mercury hm://, country/product trailers, pings 0x04).
3. **`0x0c → 0x0d` correlation.** Send `RequestKey` (raw cmd `0x0c`: file_id(20B) + track gid(16B) + a u32 sequence + zero padding), await the matching `AesKey` (`0x0d`, 16-byte key) / `AesKeyError` (`0x0e`) by sequence id, with the WaveeMusic timeout/retry discipline (2.5 s/attempt, 5 retries, reconnect after 2 timeouts, `AudioKeyManager.cs:124-258`). **This is a raw AP packet exchange, not Mercury** (`Mercury.cs` is the separate hm:// framing).
4. **Cache + reconnect.** mem → SQLite key cache; AP reconnect (DH+Shannon re-handshake) on drop, sharing the `LiveSession` reusable credential.
5. **`IAudioKeySource`** rides this. `StubAudioKeySource` (16 zero bytes) for headless.

**Sequencing risk:** until this lands, the only working path is "host derives when `Key` empty" (the deferred PlayPlay fallback), which would make the deferred host the *de facto* key source early — call this out so the schedule doesn't assume key-fetch is "wire the socket." Decision in §8.

## 6. The audio-host boundary

### 6.1 Where the seam sits (one line)
```
QueueCore.Current(Track)
   → ITrackResolver.ResolveAsync           [IN SCOPE: full-metadata file-select + IAudioKeySource(persistent AP 0x0c) key + storage-resolve CDN]
   → AudioStreamHandle
   → IAudioHost.Load + Play                 ◀── THE SEAM (after storage-resolve + key-fetch, before decrypt/decode/output)
   → { DEFERRED: open CDN stream · AES-128-CTR decrypt · Ogg/Vorbis decode · normalize/mix · WASAPI output }
   → AudioHostSignal.PositionTick / Ended
   → NowPlayingProjection → Wavee.Core.IPlaybackState → PlaybackBridge → UI
```
**Control-plane-drives-the-seam (we are the target):** a remote `play` REQUEST → `Events("hm://connect-state/v1/")` → `ConnectCommandRouter` → `PlaybackController.Play(ctx,idx)` → resolve → `IAudioHost.Load+Play`; `Reply(id, Success)` within 10 s. The host clock → `AudioHostSignal` → (a) `NowPlayingProjection` → our UI; (b) `PutStateProjection` → `ITransport.Publish` so the cluster sees our position. **We are the controller (another device is active):** the same intents take the controller's **remote-forward branch** (§5.2(i)) instead of the local host. A `transfer` **to us** → `ConnectStateAnnouncer` SetActive + resume-at-position; a `transfer` **out** (device-picker pick, §7 Stage G) → an outbound transfer command targeting the chosen `deviceId` (the wire op `IConnectDevices.TransferAsync` must issue — decision in §8).

### 6.2 The `IAudioHost` seam + the resolved-track POD it receives
**Central insight:** the existing `IAudioEngine` (`Backend/Playback.cs:108-118`) is *already* this deferral boundary — it just needs (1) `Load(Track)` reshaped to take a fully-resolved handle (so resolution moves in front, in scope), and (2) a coalesced position-clock + `Ended` report. **Reshape it in place** (don't add a second in-process audio seam — §8).
```csharp
public interface IAudioHost : IAsyncDisposable {           // reshaped from IAudioEngine
    void Load(in AudioStreamHandle stream);                // ◀ was Load(Track)
    void Play(); void Pause(); void Stop();
    void Seek(long positionMs);
    void SetVolume(double volume01);                       // realtime, host-side (buffered-PCM-independent)
    long PositionMs { get; } bool IsPlaying { get; } bool IsBuffering { get; }
    IObservable<AudioHostSignal> Signals { get; }          // ◀ new: the clock + Ended report
}

public readonly record struct AudioStreamHandle(            // pure POD — crosses the seam (and, later, a pipe)
    string TrackUri, string FileIdHex, string CdnUrl,
    ReadOnlyMemory<byte> Key,                              // 16-byte AES key (persistent-AP fetch, IN SCOPE);
                                                           //   EMPTY ⇒ "host derives it" (deferred PlayPlay/native)
    AudioFormat Format, long DurationMs, float NormalizationGainDb);   // gain = optional hint; host APPLIES it
public enum AudioFormat { OggVorbis96, OggVorbis160, OggVorbis320 }    // ↔ metadata.proto AudioFile.Format (:305)

public readonly record struct AudioHostSignal(AudioHostSignalKind Kind, long PositionMs, int ErrorCode);
public enum AudioHostSignalKind { PositionTick, Ended, Buffering, Playing, Error }
```
`AudioStreamHandle` carries exactly the in-scope resolution outputs (CDN, the fetched key, format, duration, an optional ReplayGain hint) and keeps `FileIdHex` so the host can derive its own key when `Key` is empty. `AudioHostSignal` is the boxing-free, coalesced report channel (`PositionTick` ~1 Hz is the Driven wake; `Ended` drives auto-advance).

`IAudioKeySource` (the in-scope key producer over §5.3):
```csharp
interface IAudioKeySource { Task<ReadOnlyMemory<byte>> GetKeyAsync(ReadOnlyMemory<byte> fileId, ReadOnlyMemory<byte> trackGid, CancellationToken ct); }
```

### 6.3 The silent/stub host (in scope — lets the whole pipeline run headlessly)
`SilentAudioHost : IAudioHost` reports synthetic position/`Ended` with **zero** decrypt/decode/output. **Correction:** it is *not* a "merge of `StubAudioEngine` + `FakePlaybackProvider`'s anchor-clock" — `FakePlaybackProvider` has **no** anchor clock; its 1 Hz timer does a naive `PositionMs = min(Duration, PositionMs+1000)` (`FakePlaybackProvider.cs:170`). The wall-clock anchor (`AnchorPos + (Now-AnchorWall)`) lives in the `SeekBar` UI (`SeekBar.cs:63`). So `SilentAudioHost` should *adopt that SeekBar anchor math* host-side, not copy the provider:
```csharp
sealed class SilentAudioHost : IAudioHost {
    // anchor slab: AnchorWallMs, AnchorPosMs, DurationMs, Playing, Buffering
    // PositionMs => Playing ? min(DurationMs, AnchorPosMs + (NowMs - AnchorWallMs)) : AnchorPosMs   // the SeekBar.cs:63 math
    // Load: AnchorPos=0; Duration=stream.DurationMs; Buffering true→false
    // Play: AnchorWall=Now; Playing=true     Pause: AnchorPos=PositionMs; Playing=false
    // Seek(ms): AnchorPos=ms; AnchorWall=Now
    // a 1 Hz PeriodicTimer emits PositionTick WHILE PLAYING, and Ended when PositionMs >= DurationMs
}
```
Emitting ticks *only while playing* honors the zero-frames-when-paused guardrail. With `StubTrackResolver` (synthetic handle) + `StubAudioKeySource` (16 zero bytes) + `StubCrypto` (the passthrough at `Backend/Playback.cs:122-125`, kept as the deferred-decrypt placeholder), the **entire up-to-the-boundary pipeline** — remote command → controller → resolve → host → projection → `IPlaybackState` → bridge — runs end-to-end headlessly. **Correction:** it does *not* run "under the alloc tripwire and `BackendSelfTest`" — neither exists for this backend; the gate is the *new* harness of §5.2(k), and the engine's `FluentGpu.VerticalSlice` tripwire never touches this code.

### 6.4 Responsibilities DEFERRED behind `IAudioHost` (listed, not designed)
1. **AES-128-CTR decrypt-to-PCM** (real version of `StubCrypto.Decrypt`, `Backend/Playback.cs:122-125`).
2. **Ogg/Vorbis decode** (strip the 167-byte `0xA7` header; NVorbis pure-managed *or* native `libvorbis`/`stb_vorbis` via `[LibraryImport]` — AOT-gated).
3. **Mixer / DSP** — normalization *apply* (the `NormalizationGainDb` hint), EQ, compressor, limiter.
4. **Audio output** — WASAPI/PortAudio sink on a **confined audio-device thread** (a new domain; COM per-thread-confined; publishes a coalesced position, never a per-frame write).
5. **The out-of-process + named-pipe + supervisor/heartbeat/handoff apparatus** (an *alternate* `IAudioHost` impl).
6. **PlayPlay / x86_64 native key-derivation fallback** (the empty-`Key`-on-handle path).

*The host internals above are intentionally not designed in this document.*

## 7. Staged roadmap

Sequenced for **single-thread-correct first**, each stage independently testable headlessly. Sizing is rough relative effort (S/M/L). One forward dependency is called out: the `IAudioHost` *interface reshape* (a ~20-line change + the trivial stub) lands with **Stage A** so D/E/F compile against it; the **hardened SilentAudioHost body + the end-to-end gate** is the **Stage H** capstone.

### Stage 0 — Live-session lifecycle + token refresh (the prerequisite) — **M**

**Exit:** the running app (under a flag) opens a persistent dealer with a token that survives expiry (a forced re-mint reconnects cleanly); with no live creds it falls back to the fake path unchanged.

### Stage A — Transport seam widening + the test harness — **M**

**Exit:** the harness pushes a REQUEST frame and observes it surface *with headers*; `Reply` emits a typed, escaped code; `ConnectionId` fires from a synthetic pusher frame; the GC-delta assertion holds against its stated budget; builds AOT-clean (no IL2026/IL3050).

Source arbitration as explicit state.
**Exit:** a scripted cluster push folds into the slab with the right flags; a stale cluster push **does not** revert a just-issued local command; the roster updates independent of playback suppression; `NowPlayingProjection` passes the harness contract.

### Stage E — Controller actions / transfer / outbound remote control — **L**
**Scope:** `PlaybackController` (decouple `PlaybackReducer` from `IAudioEngine`; controller owns `QueueCore` + resolver + host + projection fan-out, single serialization point) **with the local-vs-remote-forward branch (§5.2(i))** + `PlaybackService : Wavee.Core.IPlaybackPlayer` + `PutStateProjection` + transfer-in → `SetActive` (resume-at-position).
**Exit:** `PlaybackBridge`'s existing intents drive the real controller unchanged; when we are active, intents hit the local host; when another device is active, they forward as outbound commands; a `transfer`-to-us makes us active and publishes our state; `Ended` → auto-advance; ack semantics explicit (received vs succeeded — §8).

### Stage F — Track resolution + storage-resolve + key-fetch (persistent AP channel) — **L (largest)**
**Scope:** `ITrackResolver` with the **full-metadata file-id path** (gid via `Base62.Decode` → fetch `metadata.proto` `Track.file[]` → select `AudioFile` by `Format`, §5.2(h)) + **the persistent AP channel (§5.3)** (retain codec+socket from `SpotifyConnection`, read-pump, `0x0c→0x0d` correlation, cache, reconnect) behind `IAudioKeySource` + storage-resolve over `Request(Channel.Spclient, …)` → populated `AudioStreamHandle`; `StorageResolveResponse` added to `Wavee.Protocol`; headless stubs return a synthetic handle / zero key.
**Exit:** live (CLI) — a real track URI resolves to a handle with CDN + a 16-byte key (over the live AP channel) + format + duration; headless — the stub path returns a synthetic handle; key/CDN are cached; the AP channel survives a forced reconnect.

**Exit:** a full headless run — remote `play` → resolve → `SilentAudioHost` → projection → `IPlaybackState` → bridge — passes the harness GC-delta gate; `Ended` drives `Next()`; paused emits no ticks.

**Exit:** a completed play emits a correctly-shaped gabo envelope (pinned by a byte-exact fixture) and Recently Played / play counts reflect on the live account.

## 8. Open design decisions (for you to settle)

> **Decisions locked (2026-06-27).** The four genuine forks are settled:
> - **#3 → In-process `IAudioHost`.** OOP only later as an alternate impl behind the same seam; **no PlayPlay/x86_64 native key-derivation** — the persistent-AP-channel fetch is *the* key path. (So the empty-`Key`-on-handle "host derives" branch is **not** the plan; an empty key is an error/interim, not a strategy — see #1/#20.)
> - **#1 / #20 → Build the persistent AP channel now** (Stage F). Key is fetched over `0x0c/0x0d` and rides on `AudioStreamHandle.Key`.
> - **#5 → Ack-on-dispatch.** The dealer ack means *received*; failures surface later via PutState; stays inside the 10 s SLA.
> - **#18 → gabo telemetry IS in v1** (Recently Played / play counts) — adds **Stage I** (§7), couples to #11 (anti-fraud strings ported verbatim).
>
> **All other §8 items are locked to their stated _Recommended_ default**, notably: #14 replace `IAudioEngine` in place; #6 single controller-thread marshal; #7 explicit arbitration state machine; #8 widen `IPlaybackState` (`IsLoading`/`Error`/`CanSkipPrev`/`CanSkipNext`/`ActiveDevice`); #9 act on the `connection_id` null-reset; #10 one `Events("hm://")` firehose; #11 anti-fraud DeviceInfo/Capabilities/gabo strings **verbatim + byte-exact test**; #12/#23 source-gen Connect protos + full `metadata.proto`; #13 MediaClock after the `SignalSource` table; #15 resolve-fully-first; #16 normalization as a handle hint; #17 host-side volume; #24 stated Gen0 budget on the dealer parse path; #25 source-gen reply + escaped `requestId`. **Still to finalize at their stage (against the live wire):** #22 outbound-control route + active-device rule (Stage E) and #26 reconciliation params (Stage D). The numbered rationale below is retained as the record.

1. **Key-fetch vs decrypt placement at the seam (the crux).** Recommended: the **resolver fetches** the 16-byte AES key via the persistent AP channel (§5.3) and it rides on `AudioStreamHandle.Key` (the common path); an **empty `Key`** signals "host derives it" — reserving the PlayPlay/x86_64-native derivation for the deferred host. Confirm the split.
2. **The deferred audio-host items** (confirm the deferral list): (a) AES-128-CTR decrypt; (b) Ogg/Vorbis decode; (c) mixer/DSP/normalization-apply; (d) WASAPI output on a confined thread; (e) the OOP/named-pipe/supervisor apparatus; (f) the PlayPlay native key-derivation fallback.
3. **In-process vs out-of-process host.** WaveeMusic is OOP *primarily* for the x86_64 PlayPlay `Spotify.dll`, not isolation (`README.md:7-14`). Recommended: **default in-process `IAudioHost`**, OOP only as an alternate impl if PlayPlay / hard isolation becomes a real requirement.
4. **Decoder AOT posture** (couples to #3): NVorbis pure-managed (verify **no** Reflection.Emit / Expression.Compile) vs native `libvorbis`/`stb_vorbis` via `[LibraryImport]`.
5. **Ack semantics:** keep WaveeMusic's **ack-on-dispatch** (the dealer ack means *received*; an unavailable-track play still acks `Success`, failures surface later via PutState) or **ack-after-execution**. Pick one and make it explicit.
6. **Reducer/controller serialization point.** `QueueCore`/`PlaybackReducer` are not thread-safe; dealer commands arrive on the transport thread, **key/host signals on the AP read-pump + (deferred) audio-device threads**. Recommended: **marshal all of them onto the controller/UI thread** via the same `post(...)` discipline (`PlaybackBridge.cs:64-76`).
7. **Cluster arbitration.** Replace WaveeMusic's three overlapping mutable mode flags with an **explicit state machine** in the projection, or keep flags? (Recommended: state machine.)
8. **Widen the `Wavee.Core.IPlaybackState` contract** — add `IsLoading`/`Error`/`CanSkipPrev`/`CanSkipNext`/`ActiveDevice`/autoplay-EndOfContext (the bridge already stages `IsLoading`/`Error`, `PlaybackBridge.cs:36-37`; the contract lacks them, `Playback.cs:29-50`). Portable-seam change, acceptable now while only `FakePlaybackProvider` implements it. Confirm the field set.
9. **`connection_id` invalidation.** Surface the null reset on reconnect and have the announce gate **act** on it (drop the stale id) — fixing WaveeMusic's `.Where(id => id != null)` bug.
10. **One firehose, many routers.** Confirm `DealerRouter` (library/playlist/collection) and the new `ConnectService` (pusher/connect-state) both consume the single `ITransport.Events("hm://")` subscription by topic prefix.
11. **Desktop-parity / anti-fraud strings.** Port DeviceInfo/Capabilities + the gabo SDK/device/app context **verbatim**, pinned by a byte-exact test, never "cleaned." Decide the AOT-clean home of the hardware/identity fragments.
12. **Protobuf surface.** Confirm `Wavee.Protocol` vendors the Connect protos as source-gen (AOT) types: `PutStateRequest`/`Cluster`/`ClusterUpdate`/`PlayerState`/`DeviceInfo`/`Capabilities`/`SetVolumeCommand`/`TransferState`/`StorageResolveResponse`, **plus the full `metadata.proto` `Track`/`AudioFile`** the resolver needs (`metadata.proto:82,305` — already compiled, unused by the Lean projection).
13. **MediaClock sequencing.** Sequence position-as-`MediaClockMs` **after** the index-based `SignalSource` table is wired (`AnimScheduler.cs:43`, `AnimScheduler.Timeline.cs:75`), or the zero-frames-when-paused guarantee won't hold.
14. **Seam identity.** **Replace** `IAudioEngine` (rename + reshape `Load(Track)`→`Load(in AudioStreamHandle)` + add the clock/`Ended` report) rather than add `IAudioHost` alongside it. (Recommended.)
15. **Instant-start head handoff.** Port WaveeMusic's deferred head-start (begin on ~128 KB head before key/CDN complete) or resolve-fully-first? Recommended: **resolve-fully first; add head-start later.**
16. **Normalization data placement.** Carry `NormalizationGainDb` as an **optional hint** on the handle (host applies), or let the host parse ReplayGain from the decrypted head? (Recommended: hint on the handle.)
17. **Volume placement.** Realtime volume applied **host-side** via `IAudioHost.SetVolume` (buffered-PCM-independent) vs upstream. (Recommended: host-side.)
18. **gabo telemetry scope for v1.** Is play-history / Recently-Played / play-count telemetry (`EventService → gabo-receiver-service/v3/events`) in scope now? A separate `IPlaybackProjection` over the event log (`HistoryProjection` is the seed, `Backend/Playback.cs:191`); needs the anti-fraud context strings (#11).
19. **Dealer token-refresh ownership (NEW).** Who re-mints a login5 token on reconnect/expiry, and how is the transport's `accessToken` provider fed it — replacing today's captured constant (`SpotifyLibrarySync.cs:60`) so the reconnect re-invoke (`LiveDealerTransport.cs:67`) gets a fresh one? Recommended: the `LiveSession` owner (§5.2(a)) owns the `Login5Client` and the provider closure.
20. **Persistent AP connection for key-fetch (NEW).** Stand up the §5.3 channel (retain codec+socket from `SpotifyConnection`, read-pump, `0x0c` correlation, reconnect) in the running app, or use an alternative?
